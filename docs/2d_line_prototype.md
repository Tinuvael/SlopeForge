# Прототип 2D-линий Datamine

Это временный изолированный прототип для проверки ручного выбора сегментов линий на 2D-плане. Он не меняет основную модель SlopeForge, не создаёт таблицы PostgreSQL и не использует PostGIS.

## Поддерживаемые CSV Datamine

Поддерживаются реальные схемы экспорта Datamine:

- расширенная: `PID,X,Y,Z,SID,PTN,PVALUE,TYPE`;
- чистая: `PID,X,Y,Z,SID`;
- legacy-вариант: `XP,YP,ZP,PTN`, где `PTN` используется как Line ID только если нет `SID`, `LINE_ID`, `STRING`, `STRING_ID` или `POLYLINE_ID`.

Логические поля внутри прототипа:

- `X`, `Y`, `Z` — координаты точки;
- `LINE_ID` — идентификатор исходной полилинии. Для реальных CSV это обычно `SID`;
- `POINT_ORDER` — порядок точки внутри линии. Обычно это `PTN`, fallback — `PID`, затем порядок строк;
- `SOURCE_TYPE` — исходный `TYPE` Datamine;
- `PVALUE` — дополнительное исходное значение.

Обязательные поля: `X`, `Y`, `Z`, `LINE_ID`. `POINT_ORDER`, `SOURCE_TYPE` и `PVALUE` необязательны. Файл `PID,X,Y,Z,SID` импортируется без `TYPE`, `PVALUE` и `PTN`: типы будут `None`, порядок точек берётся из `PID`, линия — из `SID`.

## Разделители и кодировка

Разделитель определяется автоматически через `csv.Sniffer`. Поддерживаются запятая, точка с запятой и tab. В окне импорта можно вручную выбрать `Auto`, `comma`, `semicolon` или `tab`.

Поддерживаются `utf-8-sig` и `utf-8`. Для другой кодировки показывается понятное сообщение без traceback.

## Группировка и порядок точек

Точки группируются только по `LINE_ID`. Для реального Datamine это `SID`. Одинаковые `PTN` у разных `SID` не смешиваются.

Внутри каждой линии точки сортируются так:

1. если сопоставлен `POINT_ORDER`, прототип пытается преобразовать его к числу и сортирует по нему;
2. если значения одинаковые, сохраняется исходный порядок строк;
3. если `POINT_ORDER` отсутствует или нечисловой, используется исходный порядок строк.

Пересечение линий в плане не считается объединением. Например, `ROAD` может пересекать `TOE`, но исходные `SID` остаются отдельными линиями.

## Рабочий сценарий

1. Открыть кнопку **2D Plan Prototype** в главном окне.
2. Импортировать CSV.
3. Выбрать **Рабочий горизонт** или оставить **Все отметки**.
4. При необходимости включить **Показывать только рабочий горизонт**.
5. Нажать **Создать сегмент уступа** и следовать правой панели workflow.
6. Сохранить прототип в JSON и загрузить его при следующем запуске.


## Пошаговое создание сегмента уступа

Основной UX теперь построен как единый workflow. После импорта пользователь нажимает одну кнопку **Создать сегмент уступа** и дальше следует подсказкам в правой панели **Создание сегмента уступа**:

1. выбрать нижнюю линию;
2. нажать **Использовать эту линию**;
3. поставить маркер A — начало нижнего сегмента;
4. поставить маркер B — конец нижнего сегмента;
5. подтвердить нижний сегмент;
6. выбрать верхнюю линию;
7. поставить маркеры A и B для верхнего сегмента;
8. подтвердить верхний сегмент;
9. проверить предварительный итог `U-001`;
10. нажать **Создать уступ**;
11. в карточке созданного уступа нажать **Добавить промежуточный сегмент**, если нужны промежуточные линии.

Роли `lower_boundary`, `upper_boundary` и `intermediate_assessment` назначаются автоматически текущим этапом workflow. Пользователь больше не выбирает роль вручную. Старые отдельные кнопки «Выбрать линию», «Выбрать сегмент» и «Создать черновик» убраны из основного интерфейса.

### Состояния workflow

Workflow централизован в `ui/prototype_2d/bench_creation_controller.py`. Основные состояния: `IDLE`, `SELECT_LOWER_LINE`, `SELECT_LOWER_START`, `SELECT_LOWER_END`, `CONFIRM_LOWER`, `SELECT_UPPER_LINE`, `SELECT_UPPER_START`, `SELECT_UPPER_END`, `CONFIRM_UPPER`, `CONFIRM_BENCH`, `BENCH_CREATED`, `ADD_INTERMEDIATE_SELECT_LINE`, `ADD_INTERMEDIATE_START`, `ADD_INTERMEDIATE_END`, `ADD_INTERMEDIATE_CONFIRM`.

Когда линия подтверждена для сегмента, она становится активной. До подтверждения сегмента клики и маркеры принимаются только на этой активной линии; остальные линии остаются видимым контекстом, но сильно приглушаются. Escape не закрывает окно, а отменяет текущий незавершённый выбор: выбранную линию, первый маркер или оба неподтверждённых маркера.

### Маркеры

Начало и конец сегмента показываются круглыми маркерами `A` и `B`. Маркеры рисуются поверх линии, имеют постоянный экранный размер при zoom/pan и привязаны к проекции на активную полилинию. После первого клика при движении мыши показывается preview будущего сегмента. После второго клика подсвечивается фиксированный участок между маркерами.

## Рабочий горизонт и контекстные линии

Поле **Рабочий горизонт** содержит **Все отметки**, конкретные горизонтальные отметки и пункт **Переменные линии**, если такие линии есть.

Если выбран конкретный горизонт и флажок **Показывать только рабочий горизонт** выключен, линии выбранной отметки рисуются основным цветом, а остальные линии остаются видимыми как светло-серый полупрозрачный контекст. Если флажок включён, остальные горизонты скрываются только визуально: из состояния прототипа линии, сегменты и черновики не удаляются.

## Отметка линии

Для каждой линии считаются `z_min`, `z_max`, `z_median`, `is_horizontal` и `elevation`. Если `z_max - z_min <= 0.05 м`, линия считается горизонтальной, а `elevation` равна медианной Z. Если перепад больше, линия считается переменной по высоте, `elevation = None`, а в UI показывается диапазон вида `Z=680…715`.

## Выбор сегмента

Клик не обязан попадать точно в вершину. Координата курсора проецируется на ближайший отрезок выбранной полилинии. Прототип хранит положение начала и конца вдоль линии и отдельно сохраняет извлечённые точки сегмента, чтобы черновик оставался стабильным после повторного импорта изменённой линии.

Сейчас основной рабочий сценарий остаётся single-line: один сегмент сохраняется внутри одной `SID`. Для multi-line segment selection подготовлены поле `source_line_ids` и граф связности, но полный выбор сегмента через несколько `SID` ещё не включён, чтобы не сломать проверенный single-line workflow.

## Автосвязь концов

Экспериментальная функция **Автосвязь концов** строит временный граф связности. Исходные линии физически не объединяются, `SID` не теряется.

Связь создаётся только если выполняются правила:

- endpoint-to-endpoint;
- расстояние между концами не больше заданного допуска;
- совместим `assigned_type` или, если он пустой, `source_type`;
- одинаковая горизонтальная отметка в пределах Z tolerance;
- либо обе линии переменные по высоте и совместимы по типу.

Связь не создаётся для конец-середина, середина-середина, простого пересечения в плане, разных `TYPE` или разных горизонтов.

## Хранение JSON

По умолчанию состояние сохраняется в `SlopeForge/prototype/line_segment_prototype.json` внутри каталога `QStandardPaths.AppDataLocation`. Если Qt не вернул путь, используется fallback `~/.config/SlopeForge/prototype/line_segment_prototype.json`.

В JSON сохраняются импортированные линии, исходный `source_type`, пользовательский `assigned_type`, выбранные сегменты и черновики уступов.

## Ограничения

- Данные прототипа не являются производственными.
- PostgreSQL, Alembic и SQLAlchemy не используются.
- Нет расчёта RMS, перебора, недобора и аналитических ячеек.
- Нет автоматического связывания с взрывными событиями.
- Нет 3D-каркасов, PyVista, VTK, OpenGL и полноценного GIS.
- Принадлежность промежуточных сегментов уступу задаётся только вручную.
- Multi-line segment selection пока не реализован полностью: есть подготовка модели и диагностический граф связности.
